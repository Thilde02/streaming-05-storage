⭐ Custom Project
Dataset
My Kafka producer streams records from the file data/sales.csv.
This dataset contains individual retail sales transactions. Each record includes fields such as:

order_id

datetime

region_id

product_id

unit_price

quantity

currency_code

customer_id

payment_method

discount_code

I used the original sales dataset without modifying any of the source CSV files. The producer reads each row and sends it as a structured message into Kafka.

Kafka Messages
Each Kafka message represents a single sales transaction from sales.csv.
My producer sends:

one message per row

using the Kafka topic defined in my .env file

without a custom message key (messages use the default partitioning)

I did not modify the outgoing message fields in the producer.
All additional fields (discount_amount and is_high_value) were added later in the consumer during enrichment.

Consumer Processing
My consumer (kafka_consumer_hildebrand.py) receives each message from Kafka and performs several steps:

Receives the raw sales record from the Kafka topic.

Validates required fields using the data contract.

Enriches the message with:

subtotal

tax_amount

total

discount_amount (Phase 4)

is_high_value (Phase 5)

Logs detailed information about each message, including enrichment values and running statistics.

Writes valid records to:

a CSV file: hildebrand_high_value_sales.csv

a DuckDB database: hildebrand_high_value.duckdb

Tracks running statistics such as total sales, average sale, min, and max.

The consumer processes all available messages in the topic until it reaches the configured timeout or message limit.

Experiments
I completed two custom technical enhancements:

Phase 4 Enhancement — discount_amount
I added logic to compute a discount_amount field.
If a discount_code was present, I applied a 10% discount to the subtotal.
This field was added to the enriched message and written to both CSV and DuckDB.

Phase 5 Enhancement — is_high_value
I added a new classification field called is_high_value.
A sale is marked as high‑value when:

Code
total > 100
This allowed me to categorize orders and observe how many large transactions occurred in the stream.

Results
When I ran the producer and consumer together:

All messages were successfully consumed from Kafka.

Each message was enriched with the new fields.

The CSV file was created with the correct columns.

The DuckDB database was created with the correct schema and stored all valid records.

I verified the table contents using DuckDB queries.

The logs clearly showed the enrichment steps, discount calculations, and high‑value classification.

Everything worked end‑to‑end as expected.
